1. Executive Summary
The Enterprise Scientific Knowledge Operating System (ESKOS) is a mission-critical, deterministic platform designed to unify all fragmented scientific, operational, and commercial data within a multi-billion-dollar scientific manufacturing corporation.Modern enterprise environments suffer from an structural flaw: they treat Large Language Models (LLMs) as knowledge repositories. This architectural mistake leads to hallucinations, compliance violations, and massive synchronization lag.ESKOS completely reverses this paradigm. It positions a Deterministic Knowledge Layer (combining a native property Knowledge Graph and multi-tenant Hybrid Vector indexes) as the single source of truth. The LLM is decoupled entirely from the data layer, acting strictly as a transient runtime client used for contextual synthesis and linguistic transformation.

5. Enterprise Architecture
The ESKOS framework is designed around complete decoupling, relying on an event-driven microservices model. It separates data ingestion, structural orchestration, multi-agent evaluation, and consumption pipelines into clean, independent architectural zones.Core Structural TiersIngestion & Disruption Zone: Real-time listeners and change data capture (CDC) pipelines pull unstructured and structured assets from core operational platforms.Normalization & Extraction Pipeline: Distributed worker pools process data through optical character recognition (OCR), table extraction engines, and entity-linking pipelines.Deterministic Persistence Core: A dual-storage engine linking a property knowledge graph (Neo4j Cluster) with a multi-tenant vector database (Milvus Cluster) through shared unique entity identifiers.Agentic Orchestration Layer: A decoupled mesh of single-purpose micro-agents operating via Model Context Protocol (MCP) sidecars over a low-latency event bus.Human Governance Workspace: A web-based verification workbench enabling human experts to audit, adjust, and authorize agent outputs prior to external syndication.

8. Knowledge Flow Diagram
The lifecycle of an individual piece of enterprise intelligence moves through a deterministic sequence of state transformations managed by the Temporal workflow engine:

1. Ingest & Deconstruct: Execution: 100-500ms. The raw asset is captured, assigned a deterministic UUID v5 based on source path, and converted into structured Markdown. All structural elements like tables and asset metadata are fully preserved.
2. Ontological Alignment: Execution: 200-800ms. The Entity Parsing service maps identified elements directly against our defined corporate taxonomy. It explicitly links them to pre-existing parent nodes (such as specific Products, Standards, or Materials) inside the graph database.
3. Context Injection & Embedding Generation: Execution: 150-400ms. Every text chunk is permanently stamped with a structured metadata header containing its source path, ownership, and graph location. It is then embedded using dense semantic models (text-embedding-3-large) and sparse token counters (BM25 maps).
4. Cross-Verification & Fact Checking: Execution: 500-2000ms. The automated Verification Agent queries the property graph to check extracted assertions against known truth boundaries. It evaluates unit scaling, verifies compliance tags, and flags any deviation from existing product baselines.
5. Human Verification Workspace: Execution: Asynchronous / On-Demand. Flagged anomalies or new content updates are routed to a human governance workbench. Subject matter experts review, edit, and sign off on updates via an isolated, tracked ledger workspace.
6. Production Syndication: Execution: Real-time Event Broadcaster. Upon approval, the state transitions to VERIFIED_PUBLISHED. The engine fires Kafka events that instantly refresh all downstream vector indexes, public APIs, product pages, and discovery schemas.

9. Model Context Protocol (MCP) Architecture

The architecture uses an array of stateless, independent MCP servers wrapped around enterprise APIs and data stores, acting as secure execution sidecars for the LLM orchestration layer.

Protocol Compliance & Control Security
Isolation Pattern: LLMs never talk directly to data stores or construct unvalidated, low-level database strings. Instead, agents query the local MCP Host using strict, predefined JSON-RPC 2.0 schemas.
Tool Execution Governance: The MCP Server intercepts all tool outputs, maps them to precise types, and enforces field-level access controls before passing data back to the LLM context pool.

16. Workflow Engine
Long-running business logic and multi-agent coordination states are handled by a distributed Temporal.io workflow cluster. This architecture ensures that complex verification chains execute reliably, offering complete fault tolerance and state recovery even across system restarts or infrastructure drops.

                         [Start Workflow Execution]  
                                     │  
                                     ▼  
                         [Execute Activity: Extract]  
                                     │  
                                     ▼  
                       [Execute Activity: Graph Link]  
                                     │  
                                     ▼  
                      [Execute Activity: Verify Specs]  
                                     │  
                                     ▼  
                         { Factual Score Valid? }  
                               /          \  
                             Yes           No  
                             /              \

```python
from datetime import timedelta  
from temporalio import workflow  
from typing import Dict, Any

@workflow.defn  
class ScientificKnowledgeIngestionWorkflow:  
    @workflow.run  
    async def process_incoming_document(self, payload_details: Dict[str, Any]) -> Dict[str, Any]:  
          
        # Phase 1: Ingestion and Normalization Phase  
        normalized_markdown = await workflow.execute_activity(  
            "normalize_document_layout",  
            payload_details,  
            start_to_close_timeout=timedelta(minutes=5)  
        )  
          
        # Phase 2: Structural Extraction and Ontological Injection  
        entity_map = await workflow.execute_activity(  
            "extract_entities_and_link_graph",  
            normalized_markdown,  
            start_to_close_timeout=timedelta(minutes=10)  
        )  
          
        # Phase 3: Automated Truth Verification Verification  
        verification_report = await workflow.execute_activity(  
            "verify_facts_against_graph_bounds",  
            entity_map,  
            start_to_close_timeout=timedelta(minutes=3)  
        )  
          
        if not verification_report.get("is_factually_consistent"):  
            # Enforce immediate state isolation for non-compliant payloads  
            await workflow.execute_activity("route_to_quarantine_segment", verification_report)  
            raise ValueError("Factual consistency check failed. Isolation triggered.")  
              
        # Phase 4: Route Execution to the Human Authorization Workspace  
        human_approval_state = await workflow.execute_activity(  
            "await_human_expert_authorization",  
            entity_map,  
            start_to_close_timeout=timedelta(days=7)  
        )  
          
        # Phase 5: Production Publication and Broad Distribution  
        publication_results = await workflow.execute_activity(  
            "syndicate_verified_knowledge",  
            human_approval_state,  
            start_to_close_timeout=timedelta(minutes=15)  
        )  
          
        return publication_results  
```

20. Scalability Strategy
To handle high volumes of concurrent requests while maintaining low query latency, the persistence and processing layers employ horizontal data scaling:
Property Graph Scale Pattern: The Neo4j cluster uses a multi-shard architecture where operational domains are split into distinct regional databases, synced via an independent cluster governance engine.
Vector Engine Scaling: Vector indexes utilize cluster configurations that separate read queries from write pipelines, using memory-mapped segment storage strategies to isolate system compute nodes.
Query Layer Optimization: Frequent queries are offloaded to memory cache networks using semantic cache lookups. This matches incoming query strings against historical semantic vectors to avoid expensive recalculations for repetitive questions.

25. Risks & System Vulnerabilities
Semantic Drift within Core Ontologies: As product categories evolve over multi-year cycles, foundational relationship mappings risk losing alignment with operational reality.
Linguistic Context Saturation (Token Fatigue): Complex search requests processing extensive context inputs from vector and graph databases can overflow model context boundaries, leading to dropped processing states.
External AI Citation Blockades: Public generative web crawlers and answer engine scrapers frequently alter their ingestion schemas, which can disrupt automated visibility tracking metrics.

26. Mitigation Plan
Dynamic Taxonomy Auditing Frameworks: Enforce strict, mandatory validation workflows that require human data architecture approval every six months before structural schemas can be modified.
Rank-Based Context Reduction Pipelines: Run high-efficiency rank compression steps (like Cross-Encoder re-rankers) to prioritize and strip context inputs down to the minimum required tokens before forwarding requests to the LLM runtime.
Adaptive Proxy Scraper Arrays: Deploy distributed tracking architectures that automatically switch collection strategies when external search interfaces update their structural page schemas.

# **Developer Guide & Quickstart**

This repository contains the complete implementation of the ESKOS Agent System, spanning from Trust Firewall & Intelligent Ingestion (Phase 3), to the Knowledge Fabric (Phase 4), and Content Governance (Phase 5).

The system targets a shared-knowledge-graph, brand-tagged-entity model agreed for Borosil Scientific + Goel Scientific (`org_id` on every document/entity — see `common/brand_taxonomy.md`).

For the executable service flow, payload contracts, and local runbook, see `ARCHITECTURE.md`.

## System Components & Phases

### Phase 3: Trust Firewall & Intelligent Ingestion
The ingestion and hygiene backbone that ensures only safe, structurally sound, and non-spam data enters the system.
- **Edge Security**: JA4 Fingerprinting & Dynamic Proof of Work (Cloudflare Worker).
- **API Gateway**: Kong (mTLS, rate-limiting, RBAC).
- **Hygiene Pipeline**: Entropy Analysis, Structural Validation, Semantic Spam Heuristics, Duplicate Detection (SimHash), and Prompt Injection Detection.
- **Pre-flight Planner**: Schema validation, token/cost estimation, and RT vs. Batch routing.
- **Trust Scoring**: 4-tier source classification and weighted scoring.
- **Streaming & State**: Redpanda (Kafka-compatible) and Redis (circuit breakers, DLQ routing).

### Phase 4: Knowledge Fabric
The cognitive core of the system that structures, links, and serves data to the agents.
- **Graph Database**: Neo4j for semantic relationships and entity resolution.
- **Vector Database**: Qdrant for semantic search and multi-RAG storage.
- **Metadata Registry**: PostgreSQL for provenance and metadata tracking.
- **Knowledge Fabric Service**: Orchestrates chunking, embeddings, entity extraction, and RAG retrieval.

### Phase 5 & Agents: Content Governance & Agent Runtime
- **Content Governance**: Enforces OPA policies and ensures compliance across brands.
- **Agent Runtime**: The execution environment for the autonomous agents, integrating with the Knowledge Fabric to perform complex, multi-step reasoning.
- **Dashboard**: A React-based user interface (`eskos-dashboard`) for visualizing the knowledge graph, monitoring trust scores, and interacting with the AI agents.
